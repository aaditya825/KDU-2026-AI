# Security Hardening Requirements

## Feature Goal
Add externally visible security hardening to the existing FastAPI template so the API is safer by default in production-style environments and the auth endpoints are protected against basic abuse patterns.

This document defines observable behavior only. It should be used to generate black-box tests before implementation. It should not assume any internal function names, middleware classes, or module layout.

## Current Scope
This requirement applies to the existing API surface:
- `POST /api/v1/auth/register`
- `POST /api/v1/auth/login`
- `POST /api/v1/auth/refresh`
- `GET /api/v1/auth/me`
- `GET /api/v1/health/live`
- `GET /api/v1/health/ready`
- `GET /api/v1/version`

## In Scope
- security response headers
- rate limiting on auth-sensitive endpoints
- behavior when rate limits are exceeded
- compatibility of the hardening behavior with the current auth and health endpoints
- behavior differences driven by environment settings only when they are externally visible

## Out Of Scope
- changing the auth contract itself
- changing token formats
- adding CAPTCHA, email verification, or MFA
- changing business rules for registration or login
- internal implementation details such as middleware class names or library choice

## Security Header Requirements

### General Rule
All successful API responses from the in-scope endpoints must include a minimum baseline set of security-focused HTTP headers.

### Required Headers
The API must include these headers on in-scope responses:
- `X-Content-Type-Options: nosniff`
- `X-Frame-Options: DENY`
- `Referrer-Policy: no-referrer`
- `X-XSS-Protection: 0`

### Content Security Policy
- API responses that return JSON should include a restrictive `Content-Security-Policy` value suitable for non-HTML API responses.
- The exact value may vary, but it must be present and must be intentionally restrictive rather than permissive.

### Server Information
- Responses should not expose unnecessary framework or server-identifying headers beyond what is unavoidable in the runtime environment.

### Error Responses
- Security headers must also be present on standardized error responses for in-scope endpoints whenever the response is generated by the application layer.

## Rate Limiting Requirements

### Protected Endpoints
Rate limiting must be applied to these auth-sensitive endpoints:
- `POST /api/v1/auth/register`
- `POST /api/v1/auth/login`
- `POST /api/v1/auth/refresh`

### Non-Protected Endpoints
These endpoints must remain accessible without auth-rate-limit behavior:
- `GET /api/v1/health/live`
- `GET /api/v1/health/ready`
- `GET /api/v1/version`

### Default Limit Behavior
- The default rate limit configuration must come from environment-based settings.
- The limit must be strict enough to reduce abuse of auth-sensitive endpoints.
- The same configured limit should apply consistently unless a different limit is explicitly documented for a specific endpoint.

### Exceeded Limit Behavior
When a client exceeds the configured rate limit on an auth-sensitive endpoint:
- the API must return HTTP `429 Too Many Requests`
- the response must preserve the project’s standardized error shape
- the error response must make it clear that the failure was caused by rate limiting
- the response should include a retry-oriented hint if supported by the chosen mechanism

### Limit Isolation
- Hitting the rate limit on one protected auth endpoint must not break unrelated endpoints like health or version checks.

## Environment Behavior Requirements

### Configuration Source
- Rate limit thresholds and security hardening toggles, if configurable, must be environment-driven.
- No hardcoded production secrets or environment-specific values may be required to exercise the feature.

### Documentation Exposure
- Security hardening must not break `/docs` when docs are enabled.
- If docs are disabled by configuration, hardening behavior must not unintentionally re-enable them.

## Compatibility Requirements

### Existing Auth Flows
The hardening layer must preserve the current behavior of successful auth flows except for added response headers and enforced rate limits.

Specifically:
- registration should still succeed for valid new users when within the rate limit
- login should still return the current token response shape when within the rate limit
- refresh should still rotate tokens when within the rate limit
- `/auth/me` should still work with a valid bearer token

### Health and Version Endpoints
- health and version endpoints must continue to return successful responses without requiring auth
- health and version endpoints should include the new security headers
- health and version endpoints must not be rate-limited by the auth-sensitive rate limit

## Error Handling Requirements

### Standardized Error Shape
Any application-generated rate-limit error must follow the same standardized error response style already used by the API.

The response must include enough information to identify:
- that this is an error
- that the cause is rate limiting
- the request correlation context if the standardized error response already includes it

### Status Codes
- `429` must be used for exceeded rate limits
- existing auth and validation status codes must remain unchanged for non-rate-limit failures

## Black-Box Acceptance Criteria

### Security Headers
1. A successful `GET /api/v1/health/live` response includes the required baseline security headers.
2. A successful `GET /api/v1/version` response includes the required baseline security headers.
3. A successful `POST /api/v1/auth/login` response includes the required baseline security headers.
4. An application-generated error response from an auth endpoint still includes the required baseline security headers.

### Rate Limiting
5. Repeated valid or invalid requests to `POST /api/v1/auth/login` eventually produce HTTP `429`.
6. Repeated requests to `POST /api/v1/auth/register` eventually produce HTTP `429`.
7. Repeated requests to `POST /api/v1/auth/refresh` eventually produce HTTP `429`.
8. A `429` response from a protected auth endpoint uses the standardized API error shape.
9. A `429` response clearly indicates rate limiting as the cause.

### Non-Impacted Endpoints
10. Repeated requests to `GET /api/v1/health/live` do not trigger the auth-sensitive rate limit behavior.
11. Repeated requests to `GET /api/v1/version` do not trigger the auth-sensitive rate limit behavior.

### Existing Behavior Preservation
12. A valid login request below the configured threshold still succeeds with the existing token response shape.
13. A valid refresh request below the configured threshold still succeeds with the existing token rotation behavior.
14. A valid authenticated request to `GET /api/v1/auth/me` still succeeds and is not changed except for added headers.

## Edge Cases To Cover In Tests
- repeated invalid login attempts count toward the login rate limit
- repeated invalid refresh requests count toward the refresh rate limit
- security headers remain present on both success and error responses
- the rate-limit behavior is scoped to the protected endpoint behavior and does not break health checks

## Test Generation Guidance
When generating tests from this document:
- treat the API as a black box
- do not inspect implementation code first
- assert only externally visible behavior
- cover happy path, validation failures, auth failures, rate-limit failures, and non-impacted endpoint behavior
